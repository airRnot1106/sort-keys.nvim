1. INTRODUCTION                                       *sort-keys-introduction*

sort-keys.nvim sorts object keys and array elements in the current buffer or a
range using tree-sitter, similar to Vim's built-in |:sort| command — but it
reorders structured entries (preserving commas, comments, and nesting) instead
of plain lines.

Supported filetypes: json and jsonc. Additional languages can be registered
declaratively — see |sort-keys-handlers| and |sort-keys-language-packs|.

==============================================================================
2. COMMANDS                                               *sort-keys-commands*

                                                                    *:SortKeys*
:[range]SortKeys[!] [flags]
                Sort the innermost container under the cursor (shallow).
                With [range] (including a Visual selection), only the entries
                on the selected lines reorder; the rest stay pinned.
                Accepts trailing flags like |:sort|:

                  !         Reverse the comparison (descending).
                  i         Case-insensitive comparison.
                  n         Numeric comparison (lines without a number sort
                            as 0).
                  r/{pat}/  Compare the substring matched by the Lua pattern
                            {pat}. A malformed pattern is ignored (with a
                            warning) rather than raising.

                                                                *:DeepSortKeys*
:[range]DeepSortKeys[!] [flags]
                Like |:SortKeys| but recurses into nested containers
                post-order: children are sorted before their parent.

3. CONFIGURATION                                     *sort-keys-configuration*

Call `require("sort-keys").setup()` once during Neovim initialization. All
fields are optional. setup() is idempotent: each call replaces your options
and handler map wholesale; built-in packs are never mutated.

>lua
    require("sort-keys").setup({
      comparator = nil,   -- function(a, b, ctx) -> bool|nil
      handlers   = {},    -- map of config_name -> language-pack spec
    })
<

Fields ~

comparator (function|nil, default: nil)                 *sort-keys-comparator*
  Custom comparison. Receives the two entries `(a, b, ctx)` and returns true
  if `a` sorts before `b`, or nil to fall through to the default key
  comparison. It replaces the comparison base; the `!` flag still reverses
  the final result. Each entry exposes `entry.sort_key` (the normalized key).

>lua
    -- order by key length instead of lexicographically
    require("sort-keys").setup({
      comparator = function(a, b)
        return #a.sort_key < #b.sort_key
      end,
    })
<

handlers (table, default: {})
  User language-pack specs. See |sort-keys-handlers|.

Key normalization (decoding escapes so `"a\nb"` and the literal two-character
key compare equal) is always on; there is no toggle.

==============================================================================
4. HANDLERS                                               *sort-keys-handlers*

Languages are served by packs keyed off |'filetype'|. You can override a
built-in or register a new language from your config via
`setup({ handlers = { ... } })`. Every pack is processed by the same generic
extractor — there is no per-language builder to write.

------------------------------------------------------------------------------
                                              *sort-keys-handlers-override*
Override a field of a built-in ~

When the handlers key matches a built-in config name, only the fields you
supply are overlaid; everything else is inherited.

>lua
    -- Tweak the built-in JSON options.
    require("sort-keys").setup({
      handlers = {
        json = { options = { can_deep = false } },
      },
    })
<

------------------------------------------------------------------------------
                                            *sort-keys-handlers-new-lang*
Register a new language ~

Use a handlers key that does NOT match a built-in. `filetypes`, `options`,
and `query_text` are required; `key_normalizer` is optional.

>lua
    require("sort-keys").setup({
      handlers = {
        my_lang = {
          filetypes = { "my_lang", "ml" },
          options   = {
            can_sort_object = true,
            can_sort_array  = true,
            can_deep        = true,
            comment_aware   = true,
            parser_lang     = "my_lang",  -- defaults to the filetype name
          },
          query_text = [[
            ((container_node) @sortkeys.container
              (#set! sortkeys.kind "object"))
            ((pair_node
               key:   (_) @sortkeys.key
               value: (_) @sortkeys.value) @sortkeys.entry
              (#set! sortkeys.entry_kind "pair"))
            ((comment) @sortkeys.comment)
          ]],
          key_normalizer = function(text)
            return text:gsub('^"(.*)"$', "%1")
          end,
        },
      },
    })
<

See |sort-keys-language-packs| for the option and capture reference.

------------------------------------------------------------------------------
                                      *sort-keys-handlers-override-rules*
Override rules ~

handlers key matches a built-in config name:
  Partial override. options is deep-merged on top of the built-in;
  query_text / key_normalizer replace only if supplied.

Different key, but a filetype collides with a built-in:
  The user pack wins for that filetype.

Different key + new filetype:
  New language. filetypes, options, and query_text are all required.

Notes ~
- Calling setup() again replaces the user-handler map wholesale. Built-in
  packs are never mutated.
- There is no `builder` field. Languages whose AST the `sortkeys.*` query
  cannot express (irregular shapes) are not yet supported.

==============================================================================
5. LANGUAGE PACKS                                   *sort-keys-language-packs*

A built-in pack is a directory `lua/sort-keys/languages/<config_name>/` with a
`config.toml`, a `sort-keys.scm` query, and an optional `normalize.lua`. A user
pack supplies the same data through the handler spec fields. Both are driven by
the single generic extractor.

------------------------------------------------------------------------------
                                              *sort-keys-language-packs-options*
Options ~

  can_sort_object (boolean)   allow sorting object/map containers
  can_sort_array  (boolean)   allow sorting array/list containers
  can_deep        (boolean)   allow |:DeepSortKeys| recursion
  comment_aware   (boolean)   fold comments onto adjacent entries so they
                              travel with the entry they document
  parser_lang     (string)    tree-sitter language to parse with; defaults to
                              the filetype name. Set it to reuse another
                              grammar (jsonc reuses "json").
  query_file      (string)    query filename, default "sort-keys.scm"

The inter-entry separator, trailing-separator policy, and surrounding
whitespace are NOT configured: they are observed from the source and
reproduced on write-back, so unusual but valid layouts round-trip.

------------------------------------------------------------------------------
                                             *sort-keys-language-packs-captures*
Tree-sitter query captures ~

  Capture                Metadata                         Role ~
  @sortkeys.container    sortkeys.kind "object"|"array"   sortable container
  @sortkeys.entry        sortkeys.entry_kind              item inside
                           "pair"|"element"
  @sortkeys.key          —                                key node of a pair
  @sortkeys.value        —                                value node of a pair
                                                          (used for deep
                                                          recursion)
  @sortkeys.comment      —                                comment node, only
                                                          when comment_aware
  @sortkeys.pin          —                                pin a captured entry
                                                          (holds its slot;
                                                          permeable)
  @sortkeys.fence        —                                pin + block crossing
                                                          (order-sensitive
                                                          members: spread,
                                                          computed key, **splat)

------------------------------------------------------------------------------
                                          *sort-keys-language-packs-normalize*
Key normalization ~

`normalize.lua` returns a single `fun(text:string):string` turning a raw key
node's text into the sort key (quote stripping, escape decoding). Shared escape
decoders live in `require("sort-keys.core.key_escapes")`. Omit the file to
inherit another pack's normalizer (jsonc re-exports json's) or fall back to the
identity function. A user pack overrides it via the `key_normalizer` field.

The built-in packs in `lua/sort-keys/languages/{json,jsonc}/` are the
canonical reference.
